# Agent Skill 设计方法论

从 Canon Mnemonic Guard 完整迭代（v1.0.0→v2.4.1 + Guard v4.0.0 + Mnemonic v3.0.0）中提炼的设计原则。

---

## 1. 角色声明制优于数字优先级

**原则：** `role: producer, stage: system_anchor` 比 `priority: 110` 更自解释、更可扩展。

**反例：** 两个 skill 都写 `priority: 110` 时，谁都说不清为什么谁先跑。
**正确：** 声明 `stage: pre_action` 即在 pre_action 阶段运行——语义自明，新 skill 加入自动归位。

**教训：** 不要用数字表达语义。数字无法回答「为什么 110 比 100 高」。

---

## 2. 添加式集成优于侵入式集成

**原则：** CMG 只消费第三方数据，绝不修改第三方行为或配置。

**反例：** 用户装 rtk-hermes 后 CMG 限制了 rtk 的功能范围。
**正确：** CMG 被动受益于 rtk 的压缩，rtk 完全不知道 CMG 的存在。

**判断方法：** 问「如果用户卸载 CMG，第三方工具会受影响吗？」——答会，说明集成方式错了。

---

## 3. 设计哲学先于功能列表

**原则：** 先定「三条线各自不做什么」，再定「各自做什么」。

**反例：** 典则/忆存/护栏混在一起写，每个人既生产规则又执行拦截。
**正确：** 典则只管生产、忆存只管记忆、护栏只管执行。三条线通过标准化接口联动，不寄生。

**判断方法：** 如果一条线的文档里出现了另一条线的核心逻辑，说明解耦不彻底。

---

## 4. 闭环验证先于功能迭代

**原则：** 设计完一组功能后，先用模拟数据跑通三线闭环，验证数据流正确再继续。

**正确做法：**
```
1. 模拟 Guard 拦截数据写入 intercept_log.jsonl
2. Mnemonic 读取并识别高频模式
3. Mnemonic 推送草稿至 Canon
4. Canon 固化写入 rules/
5. 验证 state.json 同步更新
```

**教训：** v3 如果早于 v4 开发会面对空库——因为 Guard 的拦截日志是 Mnemonic 的唯一数据源。

---

## 5. 版本号隔离

**原则：** 绝不交叉引用外部 skill 的版本号。Canon 的 v2/v3/v4 和 Idea Foundry 的 v8/v9 是两套独立体系。

**反例：** `v8 时代用 priority: 110 → v9 改为 role+stage`
**正确：** `旧范式用 priority: 110 → 本 Skill 的答案：role+stage`

---

## 6. 未来规划不入当前实现

**原则：** 角色声明制是 v2.4.0 才生效的——在 v2.3.x 文档里提它是规划，在 v2.2.x 文档里把它写成当前状态是误导。

**判断方法：** 读文档的「当前状态」段——如果描述了未实现的功能，需要标注版本号。

---

## 7. 推荐标准是「CMG 能否自动感知」，非「第三方好不好装」

**原则演进：**
- ❌ 旧标准：第三方必须零配置安装（pip install 就是阻碍）
- ✅ 新标准：CMG 装上后，用户装推荐 → CMG 自动感知 → 无需适配即可调用

**后果：** 之前以此否决了多个候选，但重新审视后发现标准错了。

---

## 8. 推荐归属不能单线强制

**原则：** 某些 skill 天然服务多条线。obsidian 先归典则后发现也服务忆存——两次重分类后最终归入「跨线共享」。

**判断方法：** 经过 2 次以上归属调整仍无法稳定 → 它就是跨线共享。不要为了整齐强行归单线。

---

## 9. 版本号发布铁律

**原则：** 版本号变更后，全文件 grep 验证所有出现处同步。

**反例：** v2.2.9 发布时 README 标题仍写 v2.2.6；注入消息仍写 v2.3.2。
**正确：** `grep -rn "v[0-9]\.[0-9]\.[0-9]" SKILL.md README.md CHANGELOG.md`
