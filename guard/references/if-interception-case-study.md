# Guard 风险分级实战验证 — IF 拦截删除操作

**来源**: 2026-05-25 会话，CMG 规则库修复过程中 Guard 成功拦截了一次不可逆操作。

## 场景

Agent 准备删除 6 个 unknown 规则文件（共 33 次历史命中记录）时，Guard 检测到"删除文件"是不可逆操作。

## 拦截链

```
Agent 提出: "A: 标 monitor 保留 / B: 删掉重建"
    │
    ▼
Guard 风险分级检测: "删除文件" → 匹配不可逆关键词
    │
    ▼
Guard v4.7.1 闭环重试引擎: hard + 不可逆 → clarify 暂停确认
    │
    ▼
Clarify 弹窗: A. 标 monitor 保留 / B. 删掉重建
    │
    ▼
用户选 A → 33 次命中记录保留 ✅
```

## 验证了什么

1. **风险分级正确生效** — Guard 没有对所有 hard 规则一刀切自动重试，而是识别出"删除"为不可逆操作
2. **clarify 暂停机制正确** — 没有直接拒绝（旧 Guard），也没有自动重试（v4.7.0 盲区），而是暂停给用户选择
3. **与 Idea Foundry 的配合** — Idea Foundry 的 clarify 交互范式被 Guard 复用，用户获得了流畅的选择体验

## 关键教训

- 不可逆关键词匹配表需要持续完善。当前覆盖：删除、清空、覆盖、移除、drop、rm、自动归档改写铁则库、write_file(已存在文件)
- 这次拦截保住了 33 次历史命中记录。如果没有风险分级，v4.7.0 会自动重试（重试什么？），v4.6.0 会直接拒绝
- 证明了"删除文件必须暂停确认"这个设计决策的正确性
